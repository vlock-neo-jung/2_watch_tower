# Watch Tower - 시스템 구성도 (10대 카메라 기준)

**작성일:** 2026-02-24
**기반 문서:** [프로젝트 개요](overview.md), [PPE 감지 PoC 보고서](poc/ppe-detection.md), [기술 리서치](references/ppe-detection-technical.md)

## 1. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **Edge-Heavy** | AI 추론은 현장 Edge에서 수행. 중앙에는 이벤트+스냅샷만 전송. |
| **Pull 기반 스트리밍** | 라이브 영상은 관제 요원이 요청할 때만 WebRTC로 전송. |
| **로컬 자율 운영** | 네트워크 단절 시에도 Edge 노드가 독립적으로 감지/이벤트 생성. |
| **단계적 확장** | 10대(현재) → 20대 → 50대까지 Edge 노드 추가만으로 수평 확장. |

## 4. WebRTC 라이브 스트리밍 구성

### 4.1 프로토콜 선정

| 프로토콜 | 지연 | 대역폭/채널 | 선정 여부 |
|---------|------|------------|----------|
| **WebRTC** | **~0.5초** | **~2-4 Mbps (1080p)** | **채택** |
| HLS | 3-10초 | ~3-5 Mbps | 지연 과다 |
| RTSP 릴레이 | ~1초 | ~4-8 Mbps | 브라우저 미지원 |

### 4.2 WebRTC 시그널링 흐름

```
  Browser (대시보드)          Central API           Edge Node
       │                        │                     │
       │  1. 카메라 선택         │                     │
       │  (CAM 03 클릭)         │                     │
       │                        │                     │
       │── 2. SDP Offer ──────→ │── 3. Forward ─────→ │
       │                        │                     │
       │                        │ ←─ 4. SDP Answer ──│
       │←─ 5. Forward ─────────│                     │
       │                        │                     │
       │←─────────── 6. WebRTC Media (P2P or TURN) ──│
       │              AI 오버레이 포함 영상 스트림       │
       │              H.264, 720p, ~2Mbps             │
       │                        │                     │
       │  7. 카메라 닫기         │                     │
       │── 8. Disconnect ─────→ │── 9. Stop Stream ──→│
       │                        │                     │
```

### 4.3 네트워크 대역폭 추정

| 시나리오 | 라이브 스트림 | 이벤트 데이터 | 합계 |
|---------|-------------|-------------|------|
| 관제 미접속 (평시) | 0 Mbps | ~0.5 Mbps | **~0.5 Mbps** |
| 카메라 4대 시청 | ~8 Mbps | ~0.5 Mbps | **~8.5 Mbps** |
| 전체 10대 시청 | ~20 Mbps | ~0.5 Mbps | **~20.5 Mbps** |

> 이벤트 데이터: JSON 메타데이터(~1KB/건) + 스냅샷 JPEG(~50-100KB/건) + 영상 클립(~2-5MB/건)이 간헐적으로 발송되므로 평균 ~0.5Mbps 이하.

### 4.4 Edge 미디어 서버 구현 방식

| 방식 | 장점 | 단점 |
|------|------|------|
| **GStreamer webrtcbin** | Jetson 네이티브, HW 인코딩 | 설정 복잡 |
| mediasoup (Node.js SFU) | 웹 생태계, 확장 용이 | Edge에서 Node.js 추가 필요 |
| Janus Gateway | 성숙한 오픈소스 WebRTC 서버 | 리소스 사용량 다소 높음 |

> **권장:** GStreamer webrtcbin. Jetson의 NVENC 하드웨어 인코더를 직접 활용하여 CPU 부하 없이 H.264 WebRTC 스트림 생성 가능.

## 5. 데이터 흐름

### 5.1 상시 흐름 (이벤트 기반)

```
IP Camera ──RTSP──→ Edge Node ──HTTPS──→ Central API ──→ DB
                       │                      │
                  AI 추론/판정           WebSocket Push
                       │                      │
                  이벤트 발생 시          대시보드 알림
                  스냅샷+JSON 전송       + 알림 서비스
```

### 5.2 라이브 뷰 흐름 (요청 시)

```
Browser ──WebRTC Signaling──→ Central API ──→ Edge Node
                                                │
Browser ←─────── WebRTC Media (직접 연결) ───────┘
            AI 오버레이가 입혀진 영상 스트림
```

### 5.3 영상 보관 흐름

```
IP Camera ──RTSP──→ NVR (현장 로컬)
                    원본 영상 30일 보관
                    이벤트 시 Edge가 NVR에 클립 요청 → 중앙 전송
```

## 6. 지연시간 분석

PoC 측정 결과([PPE 감지 PoC 보고서](poc/ppe-detection.md))를 기반으로 산출.

### 6.1 End-to-End 지연 (이벤트 감지 → 알림)

```
카메라 프레임 캡처           ~10ms
HW 디코딩 (NVDEC)           ~5ms
YOLO11m TensorRT INT8 추론  ~30-50ms   ← AGX Orin 추정
                                         (MacBook CPU: 84ms)
ByteTrack 추적              ~2ms
PPE 규칙 판정               ~1ms
MediaPipe Pose (Person ROI) ~10-15ms   ← 선택적 (쓰러짐 감지 시)
이벤트 판정                  ~1ms
중앙 서버 전송 (VPN)         ~50-100ms
알림 API 호출                ~200-500ms
───────────────────────────────────
합계                        ~310-685ms → 1초 이내
```

### 6.2 Edge 노드 처리량 (AGX Orin, YOLO11m INT8)

| 구성 | 카메라당 FPS | 총 처리량 | 실시간 여부 |
|------|------------|----------|-----------|
| 5대 × 10 FPS 샘플링 | 10 | 50 FPS | O (AGX Orin ~50-70 FPS 처리 가능) |
| 5대 × 5 FPS 샘플링 | 5 | 25 FPS | O (여유) |
| 5대 × 3 FPS 샘플링 | 3 | 15 FPS | O (최소 구성) |

> **PoC 참고:** MacBook CPU에서 YOLO11m 84ms/frame (12 FPS). AGX Orin TensorRT INT8에서 2-3배 향상 예상 → 30-50ms/frame (20-33 FPS).
>
> 카메라 5대를 5-10 FPS로 샘플링하면 초당 25-50프레임. AGX Orin의 처리 능력 내에서 충분히 실시간 처리 가능.

## 7. 하드웨어 구성 (BOM)

### 7.1 현장 장비

| 구성요소 | 수량 | 사양 | 용도 | 비고 |
|---------|------|------|------|------|
| IP 카메라 | 10대 | 1080p, PoE, RTSP, IR (야간) | 영상 수집 | H.264/H.265 지원 |
| PoE 스위치 | 2대 | 8포트 이상, GbE | 카메라 전원+네트워크 | 구역당 1대 |
| **Jetson AGX Orin** | **2대** | **32GB, NVMe 256GB** | **AI 추론 (Edge)** | 각 5카메라 담당 |
| NVR | 1대 | 16TB RAID | 원본 영상 보관 (30일) | RTSP 직접 수신 |
| UPS | 2대 | 1kVA 이상 | Edge+NVR+Switch 전원 보호 | 30분 이상 버티기 |
| 산업용 케이스 | 2대 | IP65 방진방수 | Jetson 보호 | 현장 환경 대응 |
| LAN 스위치 | 1대 | 16포트 GbE | 현장 내부 백본 | Edge, NVR 연결 |

### 7.2 중앙 서버

| 구성요소 | 사양 | 용도 |
|---------|------|------|
| API/Web 서버 | 4C/16GB (클라우드 VM 가능) | FastAPI + Dashboard |
| DB 서버 | 4C/16GB + SSD 500GB | PostgreSQL + TimescaleDB |
| Object Storage | S3 호환 또는 MinIO | 스냅샷/영상 클립 저장 |

> 10대 규모에서는 API/DB를 단일 서버에 통합 가능. 50대 이상 시 분리 권장.

### 7.3 네트워크

| 구간 | 최소 대역폭 | 권장 대역폭 | 비고 |
|------|-----------|-----------|------|
| 카메라 → Edge (현장 LAN) | 각 4Mbps | 각 8Mbps | PoE 직결, 안정적 |
| 현장 → 중앙 (VPN) | 10 Mbps | 30 Mbps | 라이브 뷰 10대 동시 시 ~20Mbps |

## 8. AI 파이프라인 상세

### 8.1 선정 모델

PoC 결과([PPE 감지 PoC 보고서](poc/ppe-detection.md) 참조)에서 최종 선정.

| 항목 | 값 |
|------|------|
| 모델 | **yolo11m-construction-hazard** |
| 아키텍처 | YOLO11 Medium |
| 출처 | yihong1120/Construction-Hazard-Detection (HuggingFace) |
| 클래스 (11개) | Hardhat, Mask, NO-Hardhat, NO-Mask, NO-Safety Vest, Person, Safety Cone, Safety Vest, machinery, utility pole, vehicle |
| 파일 크기 | 39MB |
| PoC 추론 속도 | 84ms/frame (12 FPS) - MacBook M시리즈 CPU, 720p |
| 주요 감지 정확도 | Person ~90%, Hardhat ~80-88%, Safety Vest ~80-88% |
| 오탐 수준 | NO-Mask 2회/207프레임 (yolo11s 대비 91% 감소) |

### 8.2 Edge 추론 최적화

```
yolo11m-construction-hazard.pt (PyTorch, 39MB)
    │
    ▼  TensorRT 변환
yolo11m-construction-hazard.engine (TensorRT FP16/INT8)
    │
    ▼  DeepStream Pipeline
멀티스트림 동시 추론 (5대 카메라 배치)
```

| 최적화 단계 | 예상 효과 |
|------------|----------|
| TensorRT FP16 변환 | 2-3x 속도 향상 (84ms → ~30-40ms) |
| INT8 양자화 | 추가 40-60% 향상 (→ ~20-30ms) |
| DeepStream 배치 추론 | 멀티스트림 GPU 효율 극대화 |
| HW 디코딩 (NVDEC) | CPU 부하 없이 영상 디코딩 |

### 8.3 쓰러짐 감지 (향후 추가)

```
YOLO11m Person 감지
    │
    ▼ Person 바운딩 박스 크롭
MediaPipe Pose (33 keypoints)
    │
    ▼ 키포인트 좌표 분석
쓰러짐 판정 규칙
    • 어깨-엉덩이 각도 < 임계값
    • 머리 y좌표가 엉덩이 y좌표보다 낮음
    • N프레임 연속 유지
```

> MediaPipe Pose는 Apache 2.0 라이선스. Person ROI만 입력하여 처리 부하 최소화.

## 9. 이벤트 처리 흐름

### 9.1 이벤트 생성 조건

| 이벤트 유형 | 트리거 조건 | 심각도 |
|------------|-----------|--------|
| PPE 미착용 | 동일 인물 N프레임(~15프레임, ~3초) 연속 위반 | WARNING |
| 쓰러짐 감지 | 쓰러짐 자세 M프레임(~5프레임, ~1초) 연속 | **CRITICAL** |
| 위험구역 진입 | 금지구역에 Person 감지 | WARNING |

```json
  "zone": "A구역-크레인",
  "event_type": "PPE_VIOLATION",
  "severity": "WARNING",

    "violation_type": "NO-Hardhat",
    "person_track_id": 7,
    "confidence": 0.85,
    "duration_frames": 18,

  },

  "clip_url": "/events/evt_20260224.../clip.mp4"
}
```

### 9.3 알림 에스컬레이션

```
이벤트 발생
    │
    ├─ CRITICAL (쓰러짐) ──→ 즉시 카카오 알림톡 + SMS + Push
    │                        미확인 시 2분 후 재알림
    │                        5분 미확인 → 상위 관리자 알림
    │
    └─ WARNING (PPE 위반) ──→ 카카오 알림톡 + Push
                              미확인 시 5분 후 재알림
                              알림톡 실패 → SMS fallback
```

## 부록 A. 라이선스 고려사항

| 구성요소 | 라이선스 | 상용화 영향 |
|---------|---------|-----------|
| Ultralytics YOLO11 | AGPL-3.0 | Enterprise 라이선스 구매 필요 (또는 RF-DETR Apache 2.0 대안) |

| GStreamer | LGPL-2.1 | 동적 링크 시 제한 없음 |
| DeepStream | NVIDIA 무료 라이선스 | Jetson 상 무료 사용 가능 |
| ByteTrack | MIT | 제한 없음 |

> AGPL 회피가 필요한 경우 YOLO11 대신 **RF-DETR Small (Apache 2.0)** 사용 가능. PoC 시 별도 검증 필요.
