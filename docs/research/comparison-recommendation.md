### 2.2 상세 분석

#### NVIDIA DeepStream
- **장점:** GPU 가속으로 멀티스트림 처리에 최적. GStreamer 기반으로 RTSP 입력/출력 네이티브 지원. Jetson 에지 디바이스와 완벽 통합. TensorRT 자동 최적화.
- **단점:** NVIDIA GPU 종속. GStreamer 파이프라인 학습 곡선 높음. Python 바인딩이 있지만 C/C++ 기반. 디버깅 어려움.
- **적합 시점:** 5대 이상 카메라를 단일 GPU 서버에서 처리해야 할 때. Jetson 엣지 배포 시.
- **참고:** [PPE-Deepstream](https://github.com/FalconMadhab/PPE-Deepstream) 프로젝트가 건설현장 PPE 감지를 DeepStream으로 구현한 사례 있음.

#### Ultralytics YOLO
- **장점:** Watch Tower PoC에서 이미 검증 (yolo11m, yolo11s-pose). Python API가 직관적. 커뮤니티 모델 풍부. TensorRT/ONNX 내보내기 간편. HUB를 통한 클라우드 학습/배포 가능.
- **단점:** **AGPL-3.0 라이선스** - 상용 배포 시 유료 라이선스(Enterprise) 필요. 멀티카메라 파이프라인은 직접 구현 필요. 비디오 스트리밍 최적화 부족.
- **적합 시점:** PoC/파일럿 단계. 모델 학습/실험. 소규모(1-5대) 배포.
- **라이선스 비용:** Enterprise 라이선스 가격 비공개 (영업 문의). HUB Pro는 별도 구독.

#### Roboflow Supervision
- **장점:** MIT 라이선스로 상용화 자유. 바운딩박스 시각화, 추적, 존(Zone) 감지 등 유용한 유틸리티. 다양한 모델과 호환 (YOLO, SAM, DETR 등). Roboflow Universe에 건설안전 데이터셋 풍부.
- **단점:** 모델 자체가 아닌 보조 라이브러리. 파이프라인 전체를 구성해주지 않음. 대규모 비디오 처리 최적화는 별도 필요.
- **적합 시점:** 시각화, 추적, 존 감지 로직 구현 시 보조 도구로 활용. 데이터셋 구축 시.

#### Frigate NVR
- **장점:** NVR 기능 내장 (녹화, 이벤트 관리). Home Assistant 생태계. RTSP 네이티브 지원. 다양한 하드웨어 가속 (Coral, OpenVINO, ONNX).
- **단점:** **가정용 NVR** 설계 - 산업용 요구사항(알림 시스템, 대시보드, 대규모 배포)에 부적합. 커스텀 모델 통합 복잡. 건설안전 특화 기능 없음.
- **적합 시점:** Watch Tower에는 **부적합**. 참고용으로만 검토.

### 3.3 Build vs Buy 핵심 판단

**Watch Tower의 현재 상황 분석:**
1. PoC 이미 완료 - YOLO11 기반 PPE 감지 (mAP 80-88%), 쓰러짐 감지 (정상 자세 오탐 0%) 검증됨
2. 오픈소스 모델(yihong1120/Construction-Hazard-Detection)을 이미 활용 중
3. 예산 규모 미확정
4. 개인정보보호 이슈 (건설현장 영상 데이터)
5. 구역별 PPE 규칙, 장갑 감지 등 커스터마이징 필요

**권장: 시나리오 C (하이브리드) - 단계적 접근**

| 판단 요인 | 상세 |
|----------|------|
| **PoC 투자 활용** | 이미 YOLO11 기반으로 PoC가 동작하므로, 상용 SaaS로 전환하면 기존 투자 무효화 |
| **커스터마이징** | 구역별 규칙, 장갑 감지, 한국 법규 대응 등 표준 SaaS로 충족 어려움 |
| **비용 효율** | 소규모 시작 시 자체개발이 SaaS보다 비싸지만, 20대 이상에서 TCO 역전 |

## 4. 리서처 제안에 대한 비판적 평가

### 4.1 PPE 감지 리서치 평가 (research/ppe-detection.md)

#### 좋은 제안 (동의)
- **클래스별 신뢰도 임계값 조정:** NO-Mask 0.70 설정은 즉시 적용 가능하고 효과적. 반나절이면 구현 가능한 최고의 ROI 작업.
- **Temporal Smoothing (3프레임 연속 판정):** 단발성 오탐 억제에 필수적. 쓰러짐 감지에서 이미 10프레임 에피소드를 사용 중이므로 PPE에도 동일 패턴 적용 자연스러움.
- **SAHI (Slicing Aided Hyper Inference):** 원거리 소형 객체에 코드 수 줄로 AP 5-15% 향상은 매력적. 다만 3-6배 속도 저하는 실시간 파이프라인에서 심각. **매 프레임이 아닌 주기적 스캔(10초마다)에 적용**하는 것이 현실적.
- **장갑 포함 데이터셋 파인튜닝:** Construction-PPE 데이터셋(1,416장)으로 파인튜닝하면 장갑 클래스 추가 + 도메인 정확도 향상 동시 달성. **가장 근본적인 해결책으로 동의.**

#### 우려 사항 (비판)
- **RF-DETR로의 장기 전환:** Apache 2.0 라이선스의 장점은 인정하지만, 프레임워크 전환(Ultralytics → Roboflow)은 학습/배포 파이프라인 전체 재구축을 의미. 라이선스 비용과 전환 비용의 경제성 비교가 필요. Ultralytics Enterprise 라이선스가 전환 비용보다 저렴할 수 있음.
- **모델 앙상블 (YOLO + DETR):** 추론 시간 2배, GPU 메모리 2배는 멀티카메라 환경에서 하드웨어 비용 직결. 앙상블보다 **단일 고성능 모델 파인튜닝**이 비용 효율적.
- **YOLOE/YOLO-World 통합:** 오픈 보캐블러리 모델은 LVIS 35-37% AP로 고정 클래스 모델 대비 정확도가 현저히 낮음. 새 PPE 항목 프로토타이핑에만 적합하고, 프로덕션 사용은 비현실적.

#### 누락된 고려사항
- **모델 파인튜닝 데이터 품질 검증:** Construction-PPE 데이터셋 1,416장의 어노테이션 품질이 언급되지 않음. 공개 데이터셋은 노이즈 라벨이 흔하며, 이를 그대로 사용하면 파인튜닝 효과 반감.
- **야간/악천후 조건:** 건설현장은 야간 작업, 우천, 분진 등 조건이 다양한데, 이에 대한 데이터 증강 전략이 부족.

### 4.2 쓰러짐/행동 인식 리서치 평가 (research/fall-detection.md)

#### 좋은 제안 (동의)
- **키포인트 속도/가속도 특징 추가 (단기):** 추가 학습 없이 현재 규칙 엔진 확장. "수직→수평 전환 속도"로 쓰러짐 vs 앉기 구분은 직관적이고 즉시 구현 가능.
- **하이브리드 파이프라인 (규칙 1차 필터 + LSTM 2차 분류):** 점진적 구현 가능한 현실적 접근. 규칙 기반으로 99%의 정상 프레임을 빠르게 제거하고, LSTM은 의심 케이스만 처리하므로 연산 부담 최소.
- **OmniFall 데이터셋 활용:** 2025년 발표된 통합 벤치마크로 8개 데이터셋 포함. "fall" vs "sit down" vs "lie down" 등 세분화된 분류 체계가 건설현장에 유용.

#### 우려 사항 (비판)
- **RTMPose/RTMO로의 포즈 모델 전환:** COCO AP 5-8%p 향상은 매력적이지만, **두꺼운 옷/PPE 환경에서의 성능 차이는 미검증.** 연구자 본인도 "일반 COCO 벤치마크 기준"이라고 인정. RTMPose가 PPE 착용 환경에서 YOLO11-Pose보다 정말 나은지 실측이 필요.
- **MMPose 프레임워크 의존성:** RTMPose/RTMO는 MMPose 기반. 현재 Ultralytics 생태계에서 MMPose로 전환하면 학습 곡선 + 배포 복잡도가 상당히 증가. 파이프라인 리서치에서 제안한 Ultralytics model.track()과의 통합도 깨짐.
- **ST-GCN (장기):** 97%+ 정확도는 인상적이나, 건설현장 특화 데이터가 없는 상황에서 학술 벤치마크 수치를 그대로 기대하는 것은 위험. 실내 환경의 통제된 낙상 데이터와 건설현장의 실제 사고는 매우 다름.
- **액션 인식 모델 (SlowFast, X3D 등):** 연구자 스스로도 "키포인트 기반이 현재 아키텍처에 더 적합"이라고 결론 내렸으나, 문서의 1/3을 비디오 액션 인식에 할애. 현 단계에서 불필요한 조사 범위.

#### 핵심 미검증 리스크
- **건설현장 특화 낙상 데이터 부재:** 연구자도 인정했듯 "건설현장 특화 낙상 데이터셋은 공개된 것이 거의 없음." OmniFall도 실내 환경 중심. 파일럿 현장에서의 데이터 수집이 모든 ML 기반 개선의 전제 조건.
- **쪼그림/구부림 오탐:** PoC에서 가장 큰 미검증 리스크. 단기 규칙 보완으로 어느 정도 완화 가능하지만, 건설현장 연기 데이터 없이는 확신 불가.

### 4.3 실시간 파이프라인 리서치 평가 (research/realtime-pipeline.md)

#### 좋은 제안 (동의)
- **ByteTrack 추적 선택:** 고정 카메라 + 실시간 우선 환경에서 최적. Ultralytics model.track() 한 줄 적용은 최소 노력으로 최대 효과. ~1-2ms 오버헤드로 성능 영향 무시 가능.
- **프레임 교대 처리 (PPE ↔ 쓰러짐):** 두 모델을 매 프레임 순차 실행(124ms) 대신 교대 실행(84ms)으로 개선하는 전략은 우아하고 실용적.
- **5fps 샘플링:** PPE 위반은 수초간 지속되므로 5fps로 충분하다는 분석은 정확. 자원 절약 효과 큼.
- **go2rtc 기반 RTSP 프록시 + WebRTC:** 경량이고 Go 기반으로 안정적. 브라우저에서 실시간 영상 확인에 최적.
- **카카오 알림톡 + SMS 폴백 + 에스컬레이션:** 한국 시장에 맞는 현실적인 알림 체계. CoolSMS SDK 예시 코드까지 제공하여 즉시 구현 가능.

#### 우려 사항 (비판)
- **DeepStream 중규모 전환의 학습 곡선 과소평가:** "8-12주"로 추정했으나, GStreamer 파이프라인 + DeepStream 설정 + 디버깅은 실제로 더 오래 걸릴 수 있음. DeepStream은 C/C++ 기반이며, Python 바인딩의 디버깅 난이도가 높음. 팀에 DeepStream 경험자가 없다면 12-16주 이상 소요 가능.
- **Jetson Orin NX 벤치마크 신뢰도:** 인용한 벤치마크(yolo11m FP16: 15.6ms)는 640x640 기준. 실제 환경에서 전처리(디코딩, 리사이즈) + 후처리(NMS, 시각화) 포함 시 30-50% 느려질 수 있음.
- **Redis 단일 장애점:** 소규모에서 Redis Pub/Sub 사용은 적절하지만, Redis 서버 다운 시 알림 전체 실패. 건설안전에서는 알림 실패가 인명과 직결. Redis Sentinel 또는 인메모리 폴백 큐 필요.
- **Phase 1-2-3-4 로드맵이 너무 세분화:** 총 4개 Phase에 각 2-3주씩, 합산 10-15주. 현실적으로 PoC에서 파일럿까지 한 Phase로 통합하는 것이 더 효율적.

#### 누락된 고려사항
- **네트워크 불안정성:** 건설현장 Wi-Fi/LTE는 불안정. RTSP 스트림 끊김 시 자동 재연결, 프레임 손실 처리 로직이 언급되지 않음.
- **영상 녹화/저장:** 이벤트 발생 시 전후 영상 클립 저장 기능이 없음. 사후 검증과 법적 증거 확보를 위해 필수.
- **모니터링/헬스체크:** 엣지 디바이스 상태 모니터링, 모델 추론 지연 이상 감지, 카메라 상태 확인 등 운영 관점이 부족.

### 4.4 세 리서치의 교차 분석

#### 일관된 방향 (세 리서치 모두 동의)
1. **YOLO11을 기본 모델로 유지** (단기)
2. **Ultralytics 프레임워크 사용** (단기, 라이선스는 장기 해결)
3. **파인튜닝이 성능 향상의 핵심** (중기)
4. **엣지 배포 시 TensorRT 최적화** (중기)

#### 충돌하는 방향 (해결 필요)
| 주제 | PPE 리서치 | 쓰러짐 리서치 | 파이프라인 리서치 | 판정 |
|------|-----------|-------------|----------------|------|
| **포즈 모델** | 언급 없음 | RTMPose로 전환 권장 | YOLO11-Pose 유지 | **단기 YOLO11-Pose 유지, 중기 RTMPose 평가** (MMPose 전환 비용 고려) |
| **장기 모델** | RF-DETR (Apache 2.0) | ST-GCN | DeepStream nvinfer | **핵심 감지는 YOLO 파인튜닝으로 충분, 장기 아키텍처는 DeepStream** |
| **라이선스** | RF-DETR로 전환 | MMPose (Apache 2.0) | Ultralytics 유지 | **Ultralytics Enterprise 가격 확인 후 결정. 전환 비용 vs 라이선스 비용 비교** |
| **구현 복잡도** | SAHI, 앙상블 등 추가 | 하이브리드 파이프라인 | DeepStream 배치 추론 | **Phase 1에서는 모두 과잉. 단순하게 시작** |

#### 과도한 엔지니어링 경고

세 리서치를 모두 합치면 다음을 동시에 해야 함:
- 클래스별 임계값 + Temporal smoothing + SAHI + 구역별 규칙 (PPE)
- 속도/가속도 규칙 + LSTM 분류기 + RTMPose 전환 (쓰러짐)
- ByteTrack + go2rtc + Redis + FastAPI + 알림톡 + 에스컬레이션 (파이프라인)

**이 모든 것을 동시에 하면 실패한다.** 파일럿의 핵심은:
1. 실제 현장에서 동작하는지 확인
2. 실제 데이터 수집
3. 고객 피드백

이를 위해 필요한 **최소한의 것만** Phase 1에 포함해야 한다.

### 4.5 PoC 결과에 대한 종합 비판

#### PPE 감지 (yolo11m)
- **긍정:** Person ~90%, Hardhat ~80-88%, Vest ~80-88% 신뢰도는 PoC로 준수
- **우려 1:** 테스트 영상 2개(8초, 15초)는 매우 제한적. 실제 현장의 다양한 조건(야간, 우천, 분진)에서의 성능 미검증
- **우려 2:** mAP 80% 이상이라는 목표 대비, 현재는 이미지별 신뢰도만 측정. 공식 mAP 벤치마크 미수행
- **우려 3:** 장갑 감지 미지원은 프로젝트 요구사항 충족 실패. 파인튜닝으로 해결 가능하나 Phase 1에서는 "장갑 감지 = Phase 2 목표"로 명시하고 넘어가야

#### 쓰러짐 감지 (yolo11s-pose + 규칙)
- **긍정:** 정상 자세 오탐 0건은 인상적
- **심각한 우려 1:** fall_snow.mp4에서 **미감지** - 두꺼운 옷 = 건설현장 동절기 조건과 동일
- **심각한 우려 2:** 쪼그림/구부림 미테스트 - 건설현장에서 가장 흔한 작업 자세. 오탐 리스크 **매우 높음**
- **심각한 우려 3:** 하향 각도 CCTV 미검증 - 실제 건설현장 카메라는 대부분 높은 곳에서 내려다보는 각도
- **결론:** Phase 1 파일럿에서 쓰러짐 감지를 "메인 기능"으로 홍보하면 안 됨. "실험적 기능"으로 제한하고, 데이터 수집에 집중해야

## 5. 최종 추천 기술 스택

### Phase 1: 파일럿 (1-2개월) — 최소 동작 제품

**목표:** 실제 현장 1곳, 카메라 1-3대에서 핵심 기능 동작 검증

| 구성요소 | 추천 기술 | 이유 |
|---------|---------|------|
| **AI 모델 (PPE)** | YOLO11m (현재 PoC 모델 유지) | 이미 검증됨. 추가 학습 불필요 |
| **AI 모델 (쓰러짐)** | YOLO11s-Pose + 규칙 기반 (현재 유지) | 파일럿에서 실제 데이터 수집 후 개선 판단 |
| **프레임워크** | Ultralytics (AGPL, PoC 용도) | 이미 사용 중. 파일럿까지는 라이선스 이슈 없음 |
| **영상 입력** | OpenCV + RTSP | 가장 간단한 IP카메라 연동 |
| **파이프라인** | Python 단일 프로세스 | 과도한 아키텍처 피하기. multiprocessing으로 카메라별 처리 |
| **추적** | Ultralytics 내장 ByteTrack | 파이프라인 리서치 결론과 동일. 고정 카메라에서 ~1ms 오버헤드 |

| **대시보드** | Streamlit 또는 간단한 웹 UI | 빠른 프로토타이핑. 프로덕션 아님 |
| **인프라** | 현장 PC (GPU 있으면 좋지만 CPU도 가능) | MacBook 기준 1카메라 12FPS 확인됨 |

**예상 비용:** 개발 1-2명 × 1-2개월 + 현장 카메라/PC ($3,000-5,000)
**핵심 목표:** 실제 현장 데이터 수집, 오탐/미탐 측정, 고객 피드백

### Phase 2: 알파/베타 (2-4개월) — 상용화 준비

**목표:** 카메라 5-10대 지원, 안정성 확보, 상용 라이선스 정리

| 구성요소 | 추천 기술 | 이유 |
|---------|---------|------|
| **AI 모델 (PPE)** | YOLO11m + 현장 데이터 파인튜닝 | Phase 1에서 수집한 데이터로 정확도 향상 |
| **AI 모델 (쓰러짐)** | YOLO11-Pose + LSTM 시계열 분류기 | 규칙 기반의 한계 극복. 시간축 분석 추가 |
| **프레임워크** | Ultralytics Enterprise 또는 직접 ONNX Runtime | 상용 라이선스 정리 필수 |
| **영상 파이프라인** | FFmpeg + 커스텀 Python (또는 DeepStream 검토) | 멀티카메라 안정성. 5대 이상 시 DeepStream 고려 |
| **메시지 큐** | Redis Streams 또는 간단한 큐 | 카메라-감지-알림 비동기 처리 |
| **추적** | ByteTrack (Supervision 통합) | MIT 라이선스, 경량 |
| **알림** | 카카오 알림톡 + SMS Fallback | 에스컬레이션 규칙 구현 |
| **대시보드** | React + WebSocket | 실시간 모니터링 UI |
| **인프라** | 엣지 (NVIDIA Jetson AGX Orin) 또는 현장 GPU 서버 | TensorRT 최적화로 멀티스트림 |

**예상 비용:** 개발 2-3명 × 2-4개월 + Jetson ($1,000-2,000/대) + Ultralytics 라이선스
**핵심 목표:** 안정적 멀티카메라 처리, 상용 라이선스 확보, 정확도 90%+ 달성

### Phase 3: 상용 (4-6개월+) — 스케일업

**목표:** 10-50대 카메라, 다중 현장, 엔터프라이즈 기능

| 구성요소 | 추천 기술 | 이유 |
|---------|---------|------|
| **영상 파이프라인** | NVIDIA DeepStream 또는 커스텀 GStreamer | 50대 규모에서 필수 |
| **AI 추론** | TensorRT (INT8 양자화) | 처리량 극대화 |
| **아키텍처** | 엣지-클라우드 하이브리드 | 엣지에서 추론, 클라우드에서 관리/분석 |
| **메시지 브로커** | Apache Kafka 또는 NATS | 대규모 이벤트 스트리밍 |
| **저장소** | MinIO (영상) + PostgreSQL (메타데이터) | 자체 인프라에서 데이터 주권 확보 |
| **오케스트레이션** | Docker Compose → 필요 시 Kubernetes | 점진적 복잡도 증가 |
| **모니터링** | Grafana + Prometheus | 시스템 상태 모니터링 |

**예상 비용:** 개발 3-5명 × 4-6개월 + 인프라 (현장당 $5,000-15,000)

## 7. Dual-Track 제안: 최적 의사결정 방법

Phase 1 파일럿에서 자체개발과 상용 SaaS를 **동시에 평가**하는 것을 권장:

1. **자체개발 트랙:** 현재 PoC를 파일럿으로 확장 (RTSP 입력 + 알림 추가)
2. **상용 SaaS 트랙:** Spot AI 또는 Superb AI에 데모 요청, 동일 현장에서 2주간 테스트
3. **비교 기준:**
   - 감지 정확도 (PPE, 쓰러짐)
   - 커스터마이징 가능 범위 (구역별 규칙, 장갑 감지)
   - 실제 TCO (SaaS 월 비용 vs 자체개발 추가 비용)
   - 데이터 보관 방식 (개인정보보호 적합성)
   - 한국어 UI/알림 지원

이 비교 결과에 따라 Phase 2 방향을 확정하면 의사결정 리스크를 최소화할 수 있다.
