# 쓰러짐/행동 인식 기술 리서치

**작성일:** 2026-02-26
**목적:** Watch Tower 프로젝트의 낙상 감지 정확도 개선을 위한 기술 대안 조사 및 추천안 도출

---

## 1. 현재 구현 분석

### 현재 방식
- **모델:** YOLO11s-Pose (9.9M 파라미터, 19MB)
- **판정:** 규칙 기반 가중 점수 (몸통각도 w=0.4, 높이비율 w=0.3, bbox비율 w=0.3)
- **임계값:** 정규화 점수 >= 0.6이면 쓰러짐 판정
- **에피소드:** 10프레임 이상 연속 쓰러짐 시 1회 에피소드

### 확인된 문제점
| 문제 | 심각도 | 원인 |
|------|--------|------|
| 두꺼운 옷/PPE에서 키포인트 부정확 | 높음 | 포즈 추정 모델의 한계 (fall_snow 미감지) |
| 쪼그림/구부림 vs 쓰러짐 구분 미검증 | 높음 | 정적 프레임 단위 규칙만 사용 |
| 시간축 분석 부재 | 높음 | 누워서 쉬는 것 vs 쓰러지는 것 구분 불가 |
| 높은 CCTV 각도 미검증 | 높음 | 정면/측면만 테스트 |
| 가림(Occlusion) 미검증 | 중간 | 자재/장비에 의한 부분 가림 상황 |

## 2. 포즈 추정 모델 대안

### 2.1 YOLO11-Pose (현재)

| 항목 | 값 |
|------|------|
| COCO AP (val) | ~67-70% AP (s 모델 추정) |
| 추론 속도 | 40ms/frame (MacBook CPU, 720p) |
| 파라미터 | 9.9M (s), ~20M (m) |
| 특징 | 1-stage, 인물 감지+키포인트 동시 추론 |
| 라이선스 | AGPL-3.0 (상용 시 유료 라이선스 필요) |

**장점:** 단일 패스로 감지+포즈 추론, Ultralytics 생태계 활용 용이, 빠른 속도
**단점:** top-down 방식 대비 키포인트 정확도 열세, 두꺼운 옷/PPE에서 성능 저하 확인

### 2.2 RTMPose (MMPose)

| 모델 | COCO AP | CPU FPS (i7-11700) | GPU FPS (GTX 1660Ti) | 모바일 FPS |
|------|---------|---------------------|----------------------|------------|
| RTMPose-s | 72.2% | - | - | 70+ (Snapdragon 865) |
| RTMPose-m | 75.8% | 90+ | 430+ | - |
| RTMPose-l | 67.0% (WholeBody) | - | 130+ | - |

**장점:**
- YOLO11-Pose 대비 COCO AP에서 3-6%p 높은 정확도
- CPU에서도 90+ FPS로 실시간 충분
- ONNX, TensorRT, ncnn, OpenVINO 등 다양한 배포 백엔드 지원
- Jetson, ARM 등 엣지 디바이스 최적화 문서화

**단점:**
- 2-stage 방식 (인물 감지 + 포즈 추론 분리) → 별도 인물 감지기 필요
- MMPose 프레임워크 의존성 (설정 및 배포 복잡도 증가)
- Apache 2.0 라이선스 (상용 가능)

**PPE/두꺼운 옷에서의 성능:** RTMPose의 높은 AP는 일반적인 COCO 벤치마크 기준이며, 두꺼운 PPE 착용 시 성능 저하는 모든 포즈 추정 모델에 공통적으로 나타남. 다만 더 높은 기본 정확도가 열악한 환경에서도 상대적 이점을 제공할 가능성 있음.

### 2.3 RTMO (One-Stage, CVPR 2024)

| 항목 | 값 |
|------|------|
| COCO AP (val) | 74.8% (RTMO-l) |
| 추론 속도 | 141 FPS (V100 GPU) |
| 특징 | 1-stage 방식 + top-down 수준 정확도 |
| CrowdPose AP | 73.2% (혼잡 환경 SOTA) |

**장점:**
- 1-stage 방식이면서 top-down 수준의 정확도 달성
- YOLO11-Pose와 유사한 파이프라인 구조로 마이그레이션 용이
- CrowdPose에서 SOTA → 혼잡한 건설현장에서 유리

**단점:**
- 2024년 발표된 신규 모델, 커뮤니티 지원/사례 제한적
- MMPose 프레임워크 기반

### 2.4 ViTPose

| 모델 | COCO AP | 파라미터 |
|------|---------|---------|
| ViTPose-S | 73.8% | 24M |

- 대형 모델은 실시간 추론 불가 (ViTPose-G는 ~1B 파라미터)
- ViTPose-S도 YOLO-Pose 대비 느림 (2-stage 방식)

## 3. 시계열 기반 낙상 감지

### 3.1 ST-GCN (Spatial-Temporal Graph Convolutional Network)

키포인트 시퀀스를 그래프 구조로 모델링하여 공간적(관절 간 관계) + 시간적(프레임 간 변화) 특징을 동시 학습.

| 항목 | 값 |
|------|------|
| 낙상 감지 정확도 | 97.28% (최근 연구, 3-stream ST-GCN) |
| 정밀도 | 97.15% |
| F1-Score | 97.29% |
| 입력 | 키포인트 시퀀스 (e.g., 30-60 프레임) |

**장점:**
- 키포인트만 입력으로 사용 → 프라이버시 보호 (영상 자체 불필요)
- 공간-시간 관계를 동시 학습 → 쪼그림 vs 쓰러짐 구분에 효과적
- 비교적 경량 모델 (RGB 비디오 모델 대비)

**단점:**
- 키포인트 추출 품질에 의존 → 포즈 모델이 부정확하면 성능 저하
- 학습 데이터 필요 (건설현장 특화 데이터 부족)
- 구현 난이도 중간 (PyG 또는 DGL 프레임워크 필요)

**적용 가능성:** 현재 YOLO11-Pose의 키포인트 출력을 ST-GCN 입력으로 직접 활용 가능. 키포인트 버퍼(30-60 프레임)를 유지하면서 슬라이딩 윈도우 방식으로 추론.

### 3.2 LSTM/GRU 기반 키포인트 시퀀스 분류

| 항목 | 값 |
|------|------|
| 낙상 감지 정확도 | 95.2% (MediaPipe + LSTM, 2025) |
| 하이브리드 정확도 | 96.94% (CNN-LSTM) |
| 입력 | 키포인트 좌표 시퀀스 |

**장점:**
- 구현이 ST-GCN보다 단순 (표준 PyTorch/TensorFlow LSTM 레이어)
- 키포인트 속도/가속도 특징을 쉽게 추가 가능
- 경량 모델로 실시간 추론 가능

**단점:**
- 관절 간 공간적 관계를 명시적으로 모델링하지 않음
- 긴 시퀀스에서 기울기 소실 문제 가능 (GRU로 일부 완화)
- ST-GCN 대비 정확도 약간 낮음

**적용 가능성:** 가장 빠르게 구현 가능한 시계열 분석 방법. 키포인트 좌표 + 프레임 간 차분(속도) + 2차 차분(가속도)을 입력 특징으로 사용.

### 3.3 Transformer 기반 시계열 분석

| 항목 | 값 |
|------|------|
| 접근 방식 | Self-attention으로 시퀀스 내 장거리 의존성 학습 |
| 정확도 | ST-GCN과 유사하거나 약간 높음 (데이터 양에 따라) |

**장점:**
- 장거리 시간적 의존성 포착에 유리
- 병렬 처리 가능 (LSTM 대비 학습 속도 빠름)
- 최근 연구 활발

**단점:**
- 더 많은 학습 데이터 필요 (small data에서 과적합 위험)
- 모델 크기가 LSTM 대비 큼
- 건설현장 낙상 감지에 대한 검증 사례 적음

### 3.4 키포인트 속도/가속도 특징 활용 (규칙 기반 확장)

학습 모델 없이 키포인트의 시간축 변화를 규칙으로 분석하는 방법.

**추가 가능한 특징:**
- **키포인트 수직 속도:** 쓰러지는 순간 머리/어깨의 급격한 하강 감지
- **키포인트 분산 변화:** 쓰러지면서 관절이 바닥으로 모이는 패턴
- **자세 전환 속도:** 직립→수평 전환이 빠르면 쓰러짐, 느리면 자발적 동작

**장점:**
- 추가 학습 데이터 불필요
- 현재 규칙 엔진에 즉시 통합 가능
- 설명 가능성(Explainability) 높음

**단점:**
- 수동 임계값 튜닝 필요
- 복잡한 패턴 포착에 한계
- 다양한 쓰러짐 유형 커버 어려움

### 3.5 규칙 기반 + 학습 기반 하이브리드

2단계 파이프라인: 규칙 기반 1차 필터 + ML 분류기 2차 판정

| 항목 | 값 |
|------|------|
| 정확도 | 98.51% (threshold + CNN-LSTM 하이브리드, 2025 연구) |
| 민감도 | 96.70% |
| 특이도 | 99.44% |

**파이프라인 설계:**
```
프레임 입력
  → YOLO11-Pose 키포인트 추출
  → 규칙 기반 1차 필터 (현재 score >= 0.4로 낮춰서 후보 추출)
  → 키포인트 시퀀스 버퍼 (30-60 프레임)
  → LSTM/ST-GCN 2차 분류 ("쓰러짐" vs "쪼그림/앉기/눕기")
  → 최종 판정 + 알림
```

**장점:**
- 1차 필터로 대부분의 정상 프레임 빠르게 제거 → 연산 절감
- 2차 분류기가 오탐 제거 → 높은 정밀도
- 점진적 구현 가능 (규칙 먼저, ML 나중에 추가)

**단점:**
- 2단계 유지보수 복잡도
- 1차 필터 임계값을 너무 높이면 미감지, 너무 낮추면 2차 부담 증가

### 시계열 기법 비교 요약

| 기법 | 정확도 | 구현 난이도 | 데이터 필요량 | 실시간 가능 | 추천도 |
|------|--------|------------|-------------|-----------|--------|
| 속도/가속도 규칙 추가 | 중간 | 낮음 | 불필요 | O | **단기 추천** |
| LSTM/GRU 분류기 | 95-97% | 중간 | 중간 | O | **중기 추천** |
| ST-GCN | 97%+ | 높음 | 중간 | O | 장기 |
| Transformer | 97%+ | 높음 | 많음 | 조건부 | 장기 |
| 하이브리드 (규칙+ML) | 98%+ | 중간 | 중간 | O | **중기 추천** |

## 4. 액션 인식 모델 (Video-level)

전체 비디오 클립을 입력으로 받아 행동을 분류하는 방식. 키포인트 기반과 다르게 RGB 영상을 직접 처리.

### 4.1 SlowFast Networks

| 항목 | 값 |
|------|------|
| Kinetics-400 Top-1 | 79.8% (SlowFast 8x8, R101+NL) |
| 연산량 | 1.97 TFLOPs |
| 구조 | Slow pathway (의미) + Fast pathway (동작) |
| 추론 속도 | ~1초/클립 (GPU, C++ 배포 기준) |

**장점:**
- 2-stream 구조로 정적 외형 + 동적 움직임 동시 분석
- 낙상 감지에 적합한 구조 (빠른 움직임 포착)
- 잘 검증된 아키텍처, 다수의 산업 적용 사례

**단점:**
- **실시간 추론 어려움** (비디오 클립 단위 처리, ~1초/클립)
- GPU 필수, 엣지 디바이스에서 무거움
- 프라이버시 이슈 (RGB 영상 직접 처리)

### 4.2 X3D (Efficient Video Model)

| 항목 | 값 |
|------|------|
| Kinetics-400 Top-1 | 79.1% (X3D-XL) |
| 연산량 | SlowFast 대비 3-4.8x 적음 |
| 파라미터 | SlowFast 대비 5.5x 적음 |
| X3D-M 연산량 | SlowFast의 1/8.5 |

**장점:**
- SlowFast 대비 크게 경량화되면서 유사한 정확도
- 엣지 디바이스(Jetson) 배포에 더 적합
- 효율적인 3D CNN 아키텍처

**단점:**
- 여전히 비디오 클립 단위 처리 → 실시간 프레임 단위 감지와 결합 필요
- 키포인트 기반보다 무거움
- 건설현장 특화 데이터로 fine-tuning 필요

### 4.3 VideoMAE (Video Masked Autoencoder)

| 항목 | 값 |
|------|------|
| Kinetics-400 Top-1 | 87.4% (ViT-H), 90.0% (ViT-G) |
| 학습 방식 | 자기지도 사전학습 + fine-tuning |
| 특징 | 마스킹 비율 90%에서도 효과적 학습 |

**장점:**
- 최고 수준의 비디오 이해 능력
- 자기지도 사전학습 → 적은 레이블 데이터로도 fine-tuning 가능
- 데이터 효율적 학습 (적은 학습 데이터에서 유리)

**단점:**
- **매우 무거운 모델** (ViT-G는 수십억 파라미터)
- 실시간 추론 불가능 (대규모 GPU 필요)
- 엣지 디바이스 배포 비현실적

### 4.4 TimeSformer

| 항목 | 값 |
|------|------|
| Kinetics-400 Top-1 | 80.7% |
| 연산량 | 0.59 TFLOPs (SlowFast 1.97 TFLOPs 대비 1/3) |
| 추론 속도 | SlowFast 대비 ~25x 빠름 (배치 기준) |

**장점:**
- SlowFast 대비 3x 빠른 학습, 1/10 추론 연산량
- Divided Space-Time Attention으로 효율적 비디오 이해
- 긴 비디오 시퀀스 처리 가능

**단점:**
- 여전히 Transformer 기반 → 모바일/엣지 배포 도전적
- 비디오 클립 단위 처리 (프레임 단위 실시간 감지와 통합 필요)

### 액션 인식 모델 비교 요약

| 모델 | 정확도 (K400) | 연산량 | 실시간 가능 | 엣지 배포 | 추천도 |
|------|-------------|--------|-----------|---------|--------|
| SlowFast | 79.8% | 1.97T | X | 어려움 | 낮음 |
| X3D-M | ~76% | 0.23T | 조건부 | 가능 | 보통 |
| VideoMAE | 87-90% | 매우 높음 | X | X | 낮음 |
| TimeSformer | 80.7% | 0.59T | 조건부 | 어려움 | 보통 |

**결론:** 액션 인식 모델은 높은 정확도를 제공하지만, 실시간 프레임 단위 처리가 필요한 현재 파이프라인과 통합이 어려움. **키포인트 기반 시계열 분석이 현재 아키텍처에 더 적합.**

### 5.2 OmniFall (2025, 최신)

OmniFall은 2025년 발표된 통합 벤치마크로, 8개 공개 낙상 데이터셋을 통일된 10클래스 분류 체계로 정리.

**10클래스 구분:**
- 과도적 행동: fall, sit down, lie down, stand up
- 정적 상태: fallen, sitting, lying, standing

**특징:**
- 101명 피험자, 29개 카메라 뷰
- 통제된 환경 → 실제 환경(in-the-wild) 일반화 평가 프로토콜 제공
- OOPS-Fall: 실제 사고 영상으로 실환경 평가
- I3D, VideoMAE 등 사전학습 백본 실험 → 통제 환경과 실환경 간 성능 격차 확인
- HuggingFace에 공개: `simplexsigil2/omnifall`

### 5.3 건설현장 특화 데이터 현황

**현실: 건설현장 특화 낙상 데이터셋은 공개된 것이 거의 없음.**

- 기존 데이터셋은 대부분 실내 환경, 가정/요양원 낙상 중심
- 건설현장의 특수성 (PPE 착용, 높은 카메라 각도, 야외 환경, 자재/장비 가림)을 반영하는 데이터 부재
- 일부 산업안전 데이터셋(SH17 등)은 PPE 감지 중심이며 낙상 행동 분류 미포함

### 5.4 데이터 부족 해결 전략

#### 합성 데이터 생성
- **가상 환경 시뮬레이션:** Unity/Unreal Engine에서 건설현장 3D 모델 + 아바타로 낙상 장면 생성
- **Stable Diffusion 기반 증강:** 실제 현장 이미지를 기반으로 AI 합성 이미지 생성, mAP@50 92.5% 달성 사례 (2025)
- **효과:** 적은 실제 데이터에 합성 데이터 추가 시 가장 큰 성능 향상 (데이터 보충 효과)

#### 데이터 증강 파이프라인
- 좌우 반전, 소각도 회전, 포토메트릭 조정, 노이즈 추가
- 시간축 증강: 속도 변조, 프레임 드롭, 시간 역순 재생
- 키포인트 좌표 기반 증강: 좌표에 소량 노이즈 추가, 관절 각도 변조

#### 실제 현장 데이터 수집
- 파일럿 현장에서 연기(acted) 낙상 데이터 수집 (안전 매트 위)
- 정상 작업 자세(쪼그림, 구부림, 앉기, 계단 오르기 등) 대량 수집
- 자동 레이블링 파이프라인: 현재 규칙 기반 모델로 1차 레이블 → 수동 검수

### 데이터 전략 추천

| 단계 | 방법 | 예상 데이터량 | 소요 기간 |
|------|------|-------------|----------|
| 1단계 | 기존 공개 데이터셋(OmniFall) 활용 + 증강 | 수천 클립 | 1-2주 |
| 2단계 | 파일럿 현장 연기 낙상 + 정상 자세 수집 | 수백 클립 | 2-3주 |
| 3단계 | 합성 데이터 파이프라인 구축 (선택) | 수천 클립 | 3-4주 |

## 6. 현실적 접근법 평가

### 6.1 규칙 기반 즉시 보완 (단기, 1-2주)

현재 코드에 추가 학습 없이 즉시 적용 가능한 개선사항.

#### A. 키포인트 속도/가속도 특징 추가
```python
# 프레임 간 키포인트 변위 (수직 방향)
vertical_velocity = (kpts_t[NOSE][1] - kpts_t_prev[NOSE][1]) / dt
# 급격한 하강 = 쓰러짐 후보
if vertical_velocity > velocity_threshold:
    score += velocity_weight
```

#### B. 쪼그림/앉기 구분 규칙
- **무릎 각도 확인:** 쪼그림은 무릎이 접히지만 상체가 수직에 가까움
- **엉덩이-발목 거리:** 쪼그림은 엉덩이가 발목 근처로 내려감 (쓰러짐과 다름)
- **키포인트 수직 분포:** 쪼그림은 상하 압축, 쓰러짐은 좌우 확장

#### C. 자세 지속 시간 분석
- 연속 쓰러짐 에피소드 기준을 10프레임에서 조정
- "서 있다가 갑자기 수평이 된" 자세 전환 패턴 감지
- 단순 "수평 자세"와 "수직→수평 전환"을 구분

#### D. 임계값 동적 조정
- 카메라 각도에 따른 보정 계수 적용
- 인물 크기(bbox 면적)에 따른 신뢰도 가중

**예상 효과:** 오탐률 30-50% 감소, 쪼그림 오탐 대부분 해결
**구현 난이도:** 낮음 (현재 `fall_detector.py` 수정)

### 6.2 하이브리드 파이프라인 (중기, 4-8주)

#### 파이프라인 구조
```
카메라 프레임
  ├→ YOLO11-Pose (또는 RTMPose) 키포인트 추출 (~40ms)
  ├→ 규칙 기반 1차 필터 (score >= 0.4) (~1ms)
  │   ├→ 정상 → 통과
  │   └→ 의심 → 키포인트 버퍼에 추가
  └→ LSTM 2차 분류기 (30프레임 윈도우) (~10ms)
      ├→ 정상 (쪼그림/앉기) → 무시
      └→ 쓰러짐 확정 → 알림
```

#### 구현 단계
1. **데이터 수집** (2주): OmniFall + 파일럿 현장 데이터
2. **LSTM 분류기 학습** (2주): 키포인트 시퀀스 → {fall, squat, sit, stand, walk} 분류
3. **파이프라인 통합** (1-2주): 규칙 필터 + LSTM 분류기 직렬 연결
4. **평가 및 튜닝** (1-2주)

**예상 효과:** 정확도 95%+ 달성, 오탐/미탐 대폭 감소
**구현 난이도:** 중간 (PyTorch LSTM + 데이터 수집/레이블링 필요)

### 6.3 학습 기반 전환 (장기, 2-3개월)

#### 포즈 모델 업그레이드
- YOLO11-Pose → RTMPose-m 또는 RTMO-l로 교체
- COCO AP 5-8%p 향상 → PPE/두꺼운 옷에서의 키포인트 품질 개선

#### ST-GCN 분류기 도입
- LSTM 대신 ST-GCN으로 교체하여 공간-시간 동시 학습
- 건설현장 특화 데이터로 fine-tuning

#### 선택적: 액션 인식 앙상블
- X3D-M을 보조 분류기로 추가 (키포인트 기반 + RGB 기반 앙상블)
- 키포인트 기반 모델이 불확실할 때만 X3D 활성화 (연산 절약)
